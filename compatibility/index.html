COMPATIBILITY: WHAT RUNS IN IT

Everything you read runs.
Nothing you read can act.

secure-terminal has two modes. Line mode (the default) interprets no escape sequences at all: any program's output is reduced to printable ASCII, so line-oriented tools work and a hostile log cannot redraw your screen. TUI mode (opt-in, per tab) interprets escapes inside an isolated in-memory screen so full-screen programs render, while every cell is still character-filtered and the terminal still answers no query and touches no clipboard. The tables below are backed by real tests, not claims: the top Debian programs run in both modes, and every protocol, control byte and OSC code is accounted for.

How this was tested. Line mode was exercised through the shipped secure-terminal-cli wrapper; TUI mode through the exact pyte screen engine secure-terminal drives. Program popularity is the Debian popularity-contest install rank. Every "works" below means the program was run and its output or interface was captured and checked, on Debian trixie.

The trade-off, stated plainly. Because TUI mode lets a program position the cursor, a program you run can draw a misleading interface or overwrite a line you already read - so only run programs you trust in it. What it still cannot do is set the window title, read or write the clipboard, open a hyperlink escape, or smuggle an invisible character onto the screen: those stay closed in both modes.

Conformance suites

The public terminal conformance suites test how faithfully an emulator implements the VT/xterm escape repertoire. secure-terminal is deliberately not a faithful emulator in line mode - it interprets nothing on purpose - so it "fails" most conformance checks by design; that is the security property, not a bug. We track these suites for a different reason: as a corpus of every sequence a real program might emit, to confirm our stripping and our TUI subset behave. Each was reviewed for malicious content before use.

SuiteWhat it checkssecure-terminal relationship
esctest2xterm/VT escape behaviour, exhaustively, by driving a terminal and reading its replies.Reviewed safe. Automatable. Used as a sequence corpus; line mode intentionally answers none of its query probes.
pyte testsThe screen-model semantics of the engine secure-terminal uses for TUI mode.Reviewed safe. Automatable. Directly relevant: these are the semantics our TUI grid inherits.
libvterm testsA reference state machine for VT parsing and rendering.Reviewed safe. Automatable. Used to cross-check what a sequence "should" do against what we strip.
vttestThe classic interactive VT100/VT220 test; requires a human at the keyboard.Reviewed safe. Interactive, hard to automate. Used manually as a reference, not in CI.

The reviewed, automatable suites are being wired into the project's continuous-integration test run so their sequences exercise the sanitizer on every change. The security corpus of actual attacks lives separately in the terminal-poc-corpus.

Environment variables

The program you run inherits your environment, with deliberate exceptions: the size is taken from the kernel, colour is advertised as a fixed value, and every host-emulator fingerprint variable is scrubbed so a child (or a host it SSHes into) cannot learn which terminal you use or correlate your session.

Capability and reporting protocols

This is the heart of the guarantee. A terminal and the program it runs share one channel, so any reply a terminal writes back to a query lands on the program's input, exactly as if typed. secure-terminal therefore answers no query: nothing on the output path ever writes to the pseudo-terminal. That closes the whole "output becomes input" reflection / echoback / answerback CVE class by construction. The single, strictly opt-in and human-gated exception is clipboard read.

An automated test in the corpus feeds every one of these query sequences and asserts that zero bytes are ever written back to the pseudo-terminal, so the guarantee is checked, not merely asserted.

Control characters

In line mode the terminal honours only the few control bytes needed to edit the current line, drops the rest, and removes every escape sequence outright - there is no parser to attack. In TUI mode the confined screen model interprets cursor and colour escapes, but every resulting cell is still ASCII/unicode-filtered.

ByteNameLine mode
0x09Tab (HT)Passed through.
0x0ANewline (LF)Passed through (ends the line).
0x0DCarriage return (CR)Honoured - moves to column 0 (line-local).
0x08Backspace (BS)Honoured - one cell left; never crosses a line, so it cannot rewrite earlier output.
0x07Bell (BEL)Consumed - drives the bell policy; not displayed.
0x1BEscape (ESC) + any sequenceRemoved entirely. No ANSI/OSC/APC/DCS parser exists.
0x20-0x7EPrintable ASCIIShown verbatim.
0x00-0x06, 0x0B, 0x0C, 0x0E-0x1FOther C0 controlsDropped / shown as an inert _.
0x7FDelete (DEL)Dropped.
0x80-0x9FC1 controlsDropped / shown as _ (non-ASCII).

The keys you type are a tiny allowlist: printable ASCII plus Ctrl+C (SIGINT), Ctrl+Z (SIGTSTP), Ctrl+\ (SIGQUIT), Ctrl+D (EOF) and Backspace. Only the control byte is written; the kernel line discipline turns it into a signal.

OSC sequences

Operating System Command escapes are the ones that reach out of the display - the title, the clipboard, notifications, hyperlinks. Every one is a registered feature that is off by default (neutralised), individually toggleable at your own risk, and even when enabled is validated and sanitised to plain ASCII first. This closes clipboard exfiltration, hyperlink phishing, notification spoofing and title spoofing by default.
