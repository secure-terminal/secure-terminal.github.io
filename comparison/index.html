HOSTILE BYTE STREAMS: THE DIFFERENCE, MEASURED

Pipe garbage at a terminal.
Only one shrugs it off.

Untrusted bytes - random data, a program's output, a hostile log - are interpreted as escape sequences by a normal terminal, so they garble the screen, stick it in the wrong colours or character set, and can silently rewrite your window or tab title to attacker-chosen text. A reset clears the visible mess, but usually not the hijacked title. secure-terminal reduces the identical bytes to inert printable ASCII: the title is never touched, the charset shift is shown as literal text, no clipboard is written, and the only colour is its own bounded, contrast-guarded palette - never the attacker's.

The command you copy is not the command you run

The worst terminal attack needs no hostile file - only your habit of copying a command from a web page. Here is a command a site might offer you. It reads echo "test". Copy it, then reveal what actually landed on your clipboard - here, or by pasting it into output-lies' x-ray tool. It is perfectly safe to run: every part is just echo.

echo "test" 

    
  

  
You saw echo "test". Your clipboard held a whole second command chained on with
    && - echo "you did not expect this additional echo: unexpected code execution" -
    kept to a single line on purpose. Paste multiple lines and several terminals warn you: we tested it, and
    xfce4-terminal pops a "Potentially Unsafe Paste" dialog while qterminal asks "Paste multiline text?".
    One line joined by && trips neither - it slips in silently and runs the moment you press
    Enter on a line you thought said only echo "test" (go ahead and run it - both halves are just
    echo). The page only ever shows echo "test"; the Copy button's JavaScript quietly
    writes the longer command to your clipboard instead, and slips a zero-width character in right at the
    && seam - it draws nothing, so a normal terminal shows no sign it is there. Real classes,
    not hypotheticals: this is pastejacking, and the same paste channel carries
    bracketed-paste bypass
    and Trojan Source tricks.

  
secure-terminal strips the smuggling. A paste carrying any invisible, control or bidi
    character is caught before it reaches your shell: secure-terminal names exactly what is hidden in it
    ("1 invisible character") and lets you drop it to plain ASCII or cancel - so nothing
    rides along unseen and the text you are about to run is fully visible to read first. It cannot judge a visible
    command for you - no terminal can, and a plain ASCII paste still submits; that is the opt-in
    hook's job - but it guarantees there is nothing hidden inside the paste. The corpus's
    adversarial harness
    feeds every one of these payloads to secure-terminal and asserts it neutralises each one.

The "output becomes input" bug class (why "answers no query sequences" matters). A terminal and the program it runs share one two-way channel (the pseudo-terminal): the program's output comes out on it, and your keystrokes go in on it. When program output contains a query escape - say ESC[c ("what terminal are you?") - a normal terminal answers by writing the reply back onto that same channel (ESC[?1;2c). But the input side of that channel is the program's keyboard, so the reply reaches the program exactly as if you had typed it. That is the whole trick: a query hidden in attacker-controlled output makes the terminal write bytes back onto the input side, as if you had typed them - no keypress needed.

  
What it enables. One escape sets the window title or answerback string, a second reports it back, and the attacker-chosen text is injected as input on that same channel. That reflection is how a remote program reads your clipboard back through an OSC 52 query, and how a title or status report becomes spoofed input - all from output you only viewed.

  
Documented, and still recurring. David Leadbeater's 2023 review catalogued the reflection class across current terminals - including clipboard exfiltration via the OSC 52 read query - with fresh CVEs: CVE-2022-45872 (iTerm2), CVE-2022-47583 and CVE-2023-39726 (mintty), CVE-2022-23465 (SwiftTerm).

  
How secure-terminal closes the whole class - by construction. By default nothing on the output path ever writes to the pseudo-terminal, so output cannot become input - and an automated test feeds every known query sequence and asserts that zero bytes are ever written back. No device-attribute, cursor-position, version or colour query is ever answered, because no terminal-side signal can reliably tell a genuine query from a hostile one on the same channel.

  
The one exception - strictly opt-in and human-gated. Clipboard read (OSC 52) can be enabled for remote paste-over-ssh, and even then the terminal asks you once per tab before it will ever answer - with a delayed Allow button and rate-limited replies - so untrusted output in a tab you have not approved can never read your clipboard.

  
The limits: a plain multi-line ASCII paste still submits its lines - that is not deception, and no terminal treats it as such. In TUI mode a program can still draw a misleading interface within its own screen, so only run programs you trust. secure-terminal makes display and paste safe; it is not a sandbox for what you deliberately choose to run.

  
No terminal protects you from a harmful command itself. Deciding that curl x | sh (a curl bash pipe) or a subtly wrong rm is dangerous is a judgment call, not a parsing rule, so no terminal - this one included - can catch it by construction. Doing so needs something that can pause a command deemed unsafe and ask you to confirm. secure-terminal keeps that judgment out of the terminal but provides an opt-in hook: before a command runs in CLI mode it can call an external handler you configure, which judges the command and may warn, suggest a safer one, or ask you to confirm. The terminal ships no AI and no policy of its own; the logic lives in the handler. It includes two examples: a rules-based one (blocks a remote script piped into a root shell, asks on curl | sh and force-pushes) and an AI one that pipes the command to claude --print (or any model), with the command and scrollback framed as untrusted data so a hostile log cannot inject the judge. Limit: the hook sees the line you typed, not what the shell expands it to, and only in CLI mode.

  
Two forms: GUI and CLI. secure-terminal is a graphical application, and secure-terminal-cli is the same safe core as a command-line wrapper: it runs a shell or command in a pseudo-terminal and sanitizes the output before it reaches your existing terminal - so a plain console or an SSH session, exposed to the very same output attacks, gets safe display without the GUI. It sanitizes what a program displays; it does not gate your keystrokes or judge commands (that is the hook's job, below).






  
Why secure-terminal is different

  

    
A normal terminal interprets program output as commands - move the cursor, set the
      title, change the palette, shift the charset, write the clipboard. secure-terminal's default
      CLI mode interprets none of it: output is reduced to printable ASCII (plus safe,
      contrast-guarded colour), so a hostile stream has nothing to act on. See the
      compatibility page for exactly which sequences are neutralised,
      and the terminal-poc-corpus
      for the automated proofs.
