HOSTILE BYTE STREAMS: THE DIFFERENCE, MEASURED

Pipe garbage at a terminal.
Only one shrugs it off.

Untrusted bytes - random data, a program's output, a hostile log - are interpreted as escape sequences by a normal terminal, so they garble the screen, stick it in the wrong colours or character set, and can silently rewrite your window or tab title to attacker-chosen text. A reset clears the visible mess, but usually not the hijacked title. secure-terminal reduces the identical bytes to inert printable ASCII: the title is never touched, the charset shift is shown as literal text, no clipboard is written, and the only colour is its own bounded, contrast-guarded palette - never the attacker's.

What we fed them

One payload, two cases, run in nine terminal emulators and in secure-terminal, each on its own isolated headless display. Tested 2026-07-20. Each command is typed at an interactive shell, so every shot shows the prompt, the command, its output, and - tellingly - the state of the prompt after it. Both payloads and the capture harness are published so you can reproduce it.

Case A - cat /dev/random: 1,200 raw random bytes (head -c 1200 /dev/urandom), with no crafted escapes. Whatever happens is genuinely what random data does - no chosen title string is possible.

Case B - a crafted hostile log: an OSC 0 title-set to an illustrative marker (root@prod-db:~#), a stuck colour that is never reset, and a DEC line-drawing charset shift that is never reset. This is the deliberate attack a log file or program output can carry. The exact bytes are a short, readable script: hostile-script.sh.

See it: the crafted attack

The crafted log (Case B) is cat'd at a shell prompt in each terminal, captured on a real LXQt desktop (the labwc compositor with the Clearlooks window theme) on 2026-07-20. The prompt is a plain user@host - watch how the OSC 0 escape rewrites the window title to the attacker's root@prod-db:~# while the real prompt still says user@host: the title bar lies. Every terminal runs as a client of the compositor, which draws the same themed, server-side title bar on each window, so the hijacked title is legible in the bar directly - for the minimalist terminals (st, urxvt, alacritty) as much as for the ones that also draw their own menu or tool bar (konsole, the VTE terminals, secure-terminal). kitty is the one exception: it honours the OSC hijack too, but its shell integration resets the title to the cwd at the next prompt. Nothing is painted on: the title bar is the compositor's own decoration, captured as rendered. In a normal terminal the stuck colour bleeds into the returned prompt too; secure-terminal's title stays untouched, a banner flags the blocked escape, and its next prompt comes back clean. And the random stream (Case A)

head -c 1200 /dev/urandom typed at the prompt. No chosen string is possible, so any title change here is random noise, not an attack - the point is that a normal terminal still lets raw bytes corrupt its screen, while secure-terminal shows inert ASCII.

In secure-terminal's shots, each neutralised non-ASCII byte is an inert box coloured by its risk class, never by the program (so output cannot recolour or hide it):  bidi controls that reorder text,  zero-width and invisible characters,  control bytes,  other non-ASCII (look-alike / homoglyph-prone).

secure-terminal's colour. It is not colourless: safe ANSI colour is on by default (turn it off with the Colors toggle or NO_COLOR). Colour that can never conceal, and a palette a program cannot repaint.

Reproduce it yourself

Nothing here is a claim you have to take on trust. Case B is deterministic: the same bytes every run. The payload generator and the headless capture harness are in the terminal-poc-corpus repository.

An automated, always-on version ships as the opt-in terminal-resilience-tests in dist-ai: it asserts a traditional emulator's title is hijacked and that secure-terminal's output carries no escape byte and no title marker.

The "output becomes input" bug class (why "answers no query sequences" matters). A terminal and the program it runs share one two-way channel (the pseudo-terminal): the program's output comes out on it, and your keystrokes go in on it. When program output contains a query escape - say ESC[c ("what terminal are you?") - a normal terminal answers by writing the reply back onto that same channel (ESC[?1;2c). But the input side of that channel is the program's keyboard, so the reply reaches the program exactly as if you had typed it. That is the whole trick: a query hidden in attacker-controlled output makes the terminal type text into your shell, no keypress needed.

The escalation. One escape sets the window title or the answerback string, a second escape reports it back, and the reported - attacker-chosen - text, newline included, is injected and run as a command.

Not hypothetical - a real, 20-year CVE class. HD Moore documented it in 2003 (CVE-2003-0063), and it keeps recurring - David Leadbeater's 2023 review found ten more, among them CVE-2022-45872 (iTerm2), CVE-2022-47583 and CVE-2023-39726 (mintty), and CVE-2022-23465 (SwiftTerm), plus clipboard exfiltration via the OSC 52 read query.

How secure-terminal closes the whole class - by construction. By default nothing on the output path ever writes to the pseudo-terminal, so output cannot become input - and an automated test feeds every known query sequence and asserts that zero bytes are ever written back. No device-attribute, cursor-position, version or colour query is ever answered, because no terminal-side signal can reliably tell a genuine query from a hostile one on the same channel.

The one exception - strictly opt-in and human-gated. Clipboard read (OSC 52) can be enabled for remote paste-over-ssh, and even then the terminal asks you once per tab before it will ever answer - with a delayed Allow button and rate-limited replies - so untrusted output in a tab you have not approved can never read your clipboard.

The limits: a plain multi-line ASCII paste still submits its lines - that is not deception, and no terminal treats it as such. In TUI mode a program can still draw a misleading interface within its own screen, so only run programs you trust. secure-terminal makes display and paste safe; it is not a sandbox for what you deliberately choose to run.

No terminal protects you from a harmful command itself. Deciding that curl x | sh (a curl bash pipe) or a subtly wrong rm is dangerous is a judgment call, not a parsing rule, so no terminal - this one included - can catch it by construction. Doing so needs something that can pause a command deemed unsafe and ask you to confirm. secure-terminal keeps that judgment out of the terminal but provides an opt-in hook: before a command runs in CLI mode it can call an external handler you configure, which judges the command and may warn, suggest a safer one, or ask you to confirm. The terminal ships no AI and no policy of its own; the logic lives in the handler. It includes two examples: a rules-based one (blocks a remote script piped into a root shell, asks on curl | sh and force-pushes) and an AI one that pipes the command to claude --print (or any model), with the command and scrollback framed as untrusted data so a hostile log cannot inject the judge. Limit: the hook sees the line you typed, not what the shell expands it to, and only in CLI mode.

Two forms: GUI and CLI. secure-terminal is a graphical application, and secure-terminal-cli is the same safe core as a command-line wrapper: it runs a shell or command in a pseudo-terminal and sanitizes the output before it reaches your existing terminal - so a plain console or an SSH session, exposed to the very same output attacks, gets safe display without the GUI. It sanitizes what a program displays; it does not gate your keystrokes or judge commands (that is the hook's job, below).

Why secure-terminal is different

A normal terminal interprets program output as commands - move the cursor, set the title, change the palette, shift the charset, write the clipboard. secure-terminal's default CLI mode interprets none of it: output is reduced to printable ASCII (plus safe, contrast-guarded colour), so a hostile stream has nothing to act on.
