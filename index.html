Safe in both directions

Safe to paste. Safe to read.

Pasted commands are sanitized and program output is neutralized, so neither what you copy in nor what a program prints out can hide invisible characters, bidi overrides or escape sequences. secure-terminal accepts and shows plain printable ASCII by default and nothing else, so a lie has nothing to hide in whether it arrives by paste or by cat.

What is this? A terminal whose display shows only printable ASCII by default (everything else is neutralized, the way stcat does for logs) and which interprets no escape sequences from program output. Paste a command you copied from the web, or read text a program printed, and trust that what you see is what is there. When you need more, an opt-in TUI mode runs full-screen programs, ssh, vim, htop, through a restricted screen model that still filters every character.

Safe by construction

The security property is not a filter bolted on. It is what the terminal can and cannot do by default.

Output is ASCII by default

Program output is rendered as printable ASCII plus newline and tab. Bidi overrides, zero-width characters, homoglyphs, ANSI escapes and control bytes never reach the screen as anything but their neutralized form. A hostile filename in ls, a forged [ OK ] in a log, a Trojan-Source comment: none of them can redraw or reorder what you read.

No escape parser in the default mode

By default there is no ANSI parser to attack. The terminal honors no cursor moves, alternate screens or OSC hyperlinks from the child process. Parsing attacker-influenced escapes is the entire vulnerability class; a terminal that will not parse them cannot fall for them. And it has no terminal side-effects at all: no window-title control, no clipboard-write escape (OSC 52), no hyperlinks, in any mode.

Line-oriented input, with a real escape hatch

You type printable ASCII and a tiny allowlist of control keys. Nothing you paste can auto-run a hidden second line, because you see the exact bytes first. The signal keys send a real signal to the foreground process group, so Ctrl+C / Ctrl+\ bite even a raw-mode program, and a Terminate button (SIGTERM then SIGKILL) means a stuck program can never trap you.

Memory-safe

The code that reads hostile bytes is written in a memory-safe language (Python). The traditional terminal emulators, written in C, have a long history of memory-corruption bugs in exactly this parsing path; here that entire class, buffer overflows and use-after-free driven by attacker-controlled output, cannot happen by construction.

Safe, and actually usable

Safety you can live with day to day, not a museum piece.

Paste warning

Paste something with unicode or control characters and a dialog shows it two ways, side by side, the original and a reveal where every hidden character becomes a <U+XXXX> badge, before it reaches the shell. The Allow button is disabled for a few seconds so a stray Enter cannot wave a hostile paste through. A plain-ASCII paste is not interrupted.

Three unicode display modes

Per tab: Strip (non-ASCII becomes _, the safe default), Show (render legitimate unicode so you can read a log, while the invisible, bidi and homoglyph classes are still neutralized), and Reveal (every non-ASCII character as a <U+XXXX> badge, to inspect exactly what is there).

Transcripts safe to open anywhere

Save a tab's scrollback to a file. Because the buffer is already sanitized plain ASCII, the saved file is safe to open in any editor, unlike a normal terminal's raw log, which can carry the very escapes this terminal refuses.

Guaranteed exit

Signal keys reach the foreground program even in raw mode, and the Terminate action force-kills a program that ignores everything. You are never stuck inside something you cannot get out of.

TUI mode: run full-screen programs

When you need to run ssh, an editor like vim, or htop, opt in with your eyes open.

Opt-in, clearly flagged · ssh, vim, htop, tmux

The strict default cannot run curses apps. TUI mode (per tab, off by default, marked with a yellow risk indicator) interprets escapes inside an isolated in-memory screen model (pyte) that has no OS reach: it cannot set the window title or touch the clipboard, so those spoofing vectors stay closed. Every character it places is still ASCII/unicode-filtered, and colours use a contrast guard so nothing can be painted invisibly.

What you give up: a program can draw a misleading interface within its own screen, so only run programs you trust. This is "restricted-emulator safe," not "safe by construction." The default line mode, and everything the guarantee rests on, is unchanged.

Everyday comforts

The conveniences a terminal is expected to have, without loosening the model.

Tabs

Named, colour-coded, each its own shell. Per-tab theme, zoom, unicode mode and colours. Rename by double-click; a user name and a program title are kept separately.

Safe optional colours

A 16-colour subset of ANSI, off by default, with a contrast guard so text is never invisible. Honours NO_COLOR and TERM=dumb.

Session restore

Tabs and their scrollback survive a restart or reboot, under a fresh shell. Stored under ~/.local/state; on by default, and you can turn it off or clear it.

Drop-in settings

systemd-style .conf directories: a distro or admin seeds defaults in /etc/secure-terminal.d, the user overrides them. Plain KEY=value.

Install

A standard Debian source package (Debian trixie). Build it, or run it straight from a checkout.

# build and install git clone https://github.com/secure-terminal/secure-terminal cd secure-terminal dpkg-buildpackage -us -uc -b sudo dpkg -i ../secure-terminal_*.deb # or run from the checkout, no install ./usr/bin/secure-terminal

After dpkg -i, run sudo apt-get install -f to pull the dependencies (all in Debian main). Then just run secure-terminal.
