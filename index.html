A safe paste target

A terminal where paste is safe.

Copy a command into it, or text out of it, without worrying about invisible characters, bidi overrides or escape sequences. secure-terminal accepts and shows plain printable ASCII by default and nothing else, so a pasted lie has nothing to hide in.

How it stays safe See the problem it removes

What is this? A small, line-oriented terminal built in Python and Qt6. Its display shows only printable ASCII (everything else is neutralized, the way stcat does for logs), and it interprets no escape or control sequences from program output. It is a place to paste commands you copied from the web, or to read text a program printed, and trust that what you see is what is there. It is not a general-purpose terminal emulator, and that limit is the point.

Safe by construction

The security property is not a filter bolted on. It is what the terminal can and cannot do.

Output is ASCII-only, always

Program output is rendered as printable ASCII plus newline and tab. Bidi overrides, zero-width characters, homoglyphs, ANSI escapes and control bytes never reach the screen as anything but their neutralized form. A hostile filename in ls, a forged [ OK ] in a log, a Trojan-Source comment: none of them can redraw or reorder what you read.

No escape-sequence interpretation

There is no ANSI parser to attack. The terminal does not honor cursor moves, alternate screens, colors or OSC hyperlinks from the child process. Parsing attacker-influenced escapes is the entire vulnerability class; a terminal that cannot parse them cannot fall for them.

Input is a tiny allowlist

You type printable ASCII, and a small set of control keys that generate signals. Nothing you paste can auto-run a hidden second line, because the terminal is line-oriented and shows you the exact bytes first.

The keys it forwards

One-directional and minimal. The signal keys you actually reach for, and nothing that needs an escape parser.

KeySignalWhy
Ctrl+CSIGINTinterrupt the running command (the essential one)
Ctrl+ZSIGTSTPsuspend to the background (e.g. a gitk during review)
Ctrl+\SIGQUITthe "no, really quit" when Ctrl+C is ignored

What it will not do

On purpose. The omissions are the safety.

No full-screen TUIs (nano, vim, emacs, htop)

Curses applications are driven entirely by control and escape sequences, cursor moves, alternate screens. Supporting them means parsing exactly the bytes secure-terminal exists to refuse. That is probably too hard to do safely, so it is out of scope. Run those in a normal terminal, in a window you already treat as untrusted.

No non-ASCII by default

Additional character sets may become opt-in in a later version, much later, and always as an explicit allowlist rather than a general decoder. The default stays ASCII, because "safe unless you opt out" is the only default worth shipping.
